基礎用語

トークン (Token)

テキストを処理するための最小単位

BPE (Byte-Pair Encoding)

OpenAIがGPTモデルで採用したトークン化アルゴリズム:

  1. 全文字を抽出してベース語彙を構築
  2. 最も頻繁に出現する連続トークンペアを検索
  3. そのペアを1つの新しいトークンにマージ
  4. 目標語彙サイズに達するまで繰り返す
トークン消費の特性
  • 日本語は英語より多くのトークンを消費する傾向（約1.5-2倍）
  • 目安: 英語1単語 ≒ 1-2トークン、日本語1文字 ≒ 1-3トークン

プロンプト (Prompt)
システムプロンプト

LLMの動作方法や役割を定義。会話全体に適用される

ユーザープロンプト

個別リクエストごとの具体的な指示や質問

アシスタントメッセージ

LLMからの応答

入力トークン / 出力トークン
入力トークン

ユーザーが送信するプロンプトや文脈のトークン数

出力トークン

LLMが生成して返すレスポンスのトークン数

重要: 出力トークンは入力トークンより高価（通常2-5倍）
ハルシネーション (Hallucination)

事実の根拠から逸脱した不正確な出力を生成する現象

LLMは知識の限界に達しても推測で回答を続ける傾向があります。

緩和策
  • RAG（Retrieval-Augmented Generation）の導入
  • ファクトチェックプロセスの実装
  • ソース引用の明示的な要求

パラメータ

Temperature (温度)

範囲: 0.0 - 2.0（一般的にはデフォルト1.0）

  • 低温度（<1）: 保守的で予測可能な出力。高確率トークンがさらに選ばれやすい
  • 高温度（>1）: クリエイティブでランダムな出力。確率分布が平坦化

Top-K サンプリング

範囲: 1 - 語彙サイズ全体

各ステップで最も確率の高いK個のトークンに出力を制限

  • K=1: 最も確率の高いトークンのみ（greedy decoding）
  • K=50: 上位50個のトークンから選択
固定数のため、状況に応じた柔軟性は低い
Top-P サンプリング（Nucleus Sampling）

範囲: 0.0 - 1.0

累積確率に基づいてトークンを動的に選択

  • P=0.9: 上位90%の確率を占めるトークンから選択
  • Top-Kと異なり、候補トークン数が状況に応じて変動
  • 推奨開始値: 0.95
Max Tokens (最大出力トークン数)

生成される出力の上限を設定

コストと応答長のトレードオフを考慮して設定

API利用

API呼び出しの基本フロー

1. リクエスト準備
プロンプト作成、パラメータ設定
→
2. API呼び出し
HTTPリクエスト送信、認証情報付与
→
3. LLM処理
トークン化、モデル推論、次トークン予測
→
4. レスポンス生成
ストリーミングor一括送信
→
5. レスポンス受信
テキスト、使用トークン数、メタデータ
→
6. 後処理
デコード、Function Calling結果処理
ストリーミング (Streaming)

LLMが応答全体を完了する前にトークンを逐次送信する技術

  • Server-Sent Events (SSE) を使用
  • content-type: text/event-stream ヘッダー
利点
  • 待ち時間の体感短縮
  • UX向上（リアルタイムフィードバック）
Function Calling / Tool Use

LLMを外部ツール、API、関数と接続する機能

実行フロー
  1. ユーザークエリ送信
  2. LLMが適切な関数を特定、JSON形式でパラメータ出力
  3. アプリケーション側で関数実行
  4. 結果をLLMに返して最終応答生成
注: LLM自身は関数を実行しません。関数の特定とパラメータ生成のみを行います。

RAG・Fine-tuning

RAG (Retrieval-Augmented Generation)

LLMに企業独自の信頼できるデータを注入し、より正確な応答を生成するフレームワーク

仕組みフロー
1. ユーザーが質問
→
2. クエリに関連する情報を内部ソースから取得
→
3. 拡張プロンプトを作成
→
4. LLMに供給して回答を生成
利点
  • 再トレーニング不要
  • リアルタイムデータ更新可能
  • 実装コストが比較的低い
制限
ハルシネーションは完全には排除できない。検索品質に依存する。
Fine-tuning

事前トレーニングされたLLMを特定タスク/ドメインに適応させる追加トレーニング

特徴
  • 静的な知識に依存
  • 新情報には再トレーニング必要
  • モデルの振る舞いを根本から変更可能
ユースケース
  • 特定ドメインの専門用語の理解
  • 一貫したスタイル/トーンの維持
  • 特定フォーマットでの出力

RAG vs Fine-tuning 比較表

項目 RAG Fine-tuning
データ更新 リアルタイム可能 再トレーニング必要
コスト 検索インフラ必要 トレーニング費用
精度 検索品質に依存 ドメイン特化で高精度
導入難易度 中程度 高い
推奨用途 最新情報が必要な場合 スタイル/ドメイン特化

用語集

基本概念

BPE (Byte-Pair Encoding) トークン化アルゴリズム。頻出する文字列を統合してトークンを生成。
Context Window LLMが処理可能な最大トークン数。入力と出力の合計値。
Fine-tuning 特定タスクへのモデル適応。追加トレーニングにより振る舞いを変更。
Hallucination 事実に基づかない出力の生成。LLMの知識限界を超えた場合に発生。
LLM (Large Language Model) 大規模言語モデル。膨大なテキストデータで訓練されたAIモデル。
Prompt LLMへの指示・入力テキスト。システム、ユーザー、アシスタントの種類がある。
RAG (Retrieval-Augmented Generation) 検索拡張生成。外部データを取得してLLMに供給する手法。
Token テキスト処理の最小単位。単語の一部、単語全体、記号などを表す。

パラメータ

Max Tokens 最大出力トークン数。生成される応答の長さの上限。
Temperature 出力のランダム性制御パラメータ。低い値で保守的、高い値で創造的。
Top-K 上位K個のトークンに制限するサンプリング手法。固定数の候補から選択。
Top-P (Nucleus Sampling) 累積確率に基づくサンプリング。動的に候補トークン数を調整。

API・技術

API Key APIアクセス用の認証キー。リクエストの識別と課金に使用。
Batch Processing 非同期一括処理。リアルタイム性不要な大量リクエストに適用。
Embedding テキストのベクトル表現。意味的な類似性の計算に使用。
Function Calling LLMから外部関数を呼び出す仕組み。ツールやAPIとの統合に使用。
JSON-RPC JSONベースの遠隔手続き呼び出しプロトコル。軽量なAPI通信に使用。
Rate Limit APIリクエスト頻度の上限。1分あたり、1日あたりなどで設定。
SSE (Server-Sent Events) サーバーからクライアントへの単方向イベントストリーム技術。
Streaming トークンの逐次送信技術。応答完了前に部分的な結果を表示。
